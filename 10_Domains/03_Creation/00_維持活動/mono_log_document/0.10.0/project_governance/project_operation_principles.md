# **プロジェクト運用原則 (Project Operation Principles) v1.0**

> **文書名**: project_operation_principles.md (プロジェクト運用原則)
> **ドキュメント位置**: `docs/project_governance` (プロジェクト憲章等)

## **1\. 本書の目的**

本書は、『モノログ。』プロジェクトにおける、人間（プロジェクトオーナー）と複数のAIアシスタントが協調して開発を進めるための、具体的なワークフロー、役割分担、および情報管理の原則を定める。本書は、『プロジェクト憲章』に定められた理念を実現するための、実務的な運用ガイドラインである。

## **2\. 解決すべき課題：AI間の情報分断と非効率な指示**

本プロジェクトは、各分野で最高の能力を持つ複数の専門AIを協調させることで、個人開発の生産性を最大化することを目指す。しかし、各AIは独立して動作するため、放置すれば以下の問題が発生する。

* **情報のサイロ化:** 各AIの知見が分断され、人間が手動での情報伝達役となり、生産性が低下する。  
* **ちぐはぐな指示:** コードを完全に参照できない戦略AIが具体的な実装案を出すことで、現状のコードベースと食い違った指示が生まれ、実装AIが混乱し、手戻りが発生する。

本原則は、これらの問題を解決するためのものである。

## **3\. 背景：本原則が成立した経緯と制約**

本原則は、現状の技術的・プラットフォーム的制約下で生産性を最大化するための、現実的な最適解である。制約が解消された未来では、本原則も改訂されるべきである。

### **3.1 なぜ、二種類のAIを使い分けるのか？（能力の最適化）**

* **戦略・分析AI (Gemini):** 戦略立案、市場分析、アイデアの壁打ちなど、**思考の水平展開と構造化**に優れる。  
* **コーディングAI (Claude):** 大規模なソースコードの読解、長大な仕様書に基づいた実装など、**特定の文脈における深い思考と実装**に優れる。

### **3.2 なぜ、このような複雑な連携が必要なのか？（現状の制約）**

* **プラットフォーム的制約（最重要）:**  
  * **戦略・分析AI (Gemini):** Google One プレミアムプラン（定額）で利用しているが、**現時点ではWeb UI上での対話に限定されている。** これにより、以下の制約が発生する。  
    * ローカルのプロジェクトファイルに直接アクセスできない。  
    * 一度に読み込める情報量（コンテキストウィンドウ）に限界がある。  
    * 結果として、現状のコードベースを理解させるためには、手動での情報入力や『AIブリーフィング』といった要約ドキュメントが不可欠となる。  
  * **コーディングAI (Claude):** こちらも定額プランだが、より大きなコンテキストウィンドウを持ち、開発ツールとの連携が比較的容易であるため、実装フェーズに適している。

### **3.3 制約がない場合の理想像**

単一の、あるいは複数のAIが、プランの種類（定額 or 従量課金）に関わらず、APIやターミナル経由でプロジェクトの全情報に直接アクセスできる状態。本原則は、この理想に至る過渡期における最適化戦略である。

## **4\. 登場人物と役割定義**

| 役割 | 担当 | 主な責務と注意事項 |
| :---- | :---- | :---- |
| **プロジェクトオーナー** | あなた様 | プロジェクト全体の最終的な**意思決定者**。AIが提示する戦略や実装選択肢を評価し、「判断」を下すことに集中する。 |
| **戦略・分析AI** | Gemini | プロジェクトの\*\*「Why（なぜやるのか）」**、**「What（何をやるのか）」**、**「Design（どう見せるか）」\*\*を定義する戦略家。\<br\> 【最重要禁止事項】 原則として具体的な実装方法（How）には一切言及しない。 |
| コーディングAI | Claude | プロジェクトの\*\*「How（どうやってやるのか）」\*\*を計画・実行する実装担当。現状のコードベースに基づいた、現実的な実装選択肢を提示する。 |

## **5\. 協業ワークフロー**

仕様策定から実装までのプロセスを、以下の6階層で定義する。

| フェーズ | 内容 | 担当 | 主要な成果物 |
| :---- | :---- | :---- | :---- |
| **第1層：戦略 (Why)** | なぜ作るのか？ | 戦略・分析AI | ユーザーの物語 |
| **第2層：要件 (What)** | 何を達成すべきか？ | 戦略・分析AI | 受け入れ条件 |
| **第3層：設計 (Design)** | どう見せ、どう使わせるか？ | 戦略・分析AI | **仕様書ドラフト** |
| **第4層：レビューと承認** | 設計内容は妥当か？ | **プロジェクトオーナー** | **承認済み仕様書** |
| **第5層：実装計画 (Planning)** | どう作るか？影響範囲は？ | コーディングAI | **承認済み実装計画書** |
| **第6層：実装 (Code)** | 計画に基づきコードを記述する | コーディングAI | ソースコード, 実装ドキュメント |

## **6\. 各フェーズの詳細**

### **第1層〜第3層：仕様書ドラフトの作成**

1. **現状の共有（ブリーフィング）:** (任意) プロジェクトオーナーは、コーディングAIに依頼し、最新の実装状況を要約した\*\*『AIブリーフィング』\*\*を作成させ、戦略・分析AIにインプットする。  
2. **戦略・要件・設計の策定:** 戦略・分析AIが主体となり、「戦略」「要件」「設計」を担当し、それらを統合した成果物として\*\*『仕様書ドラフト』\*\*を作成する。プロジェクトオーナーはゼロから考える必要はない。

### **第4層：レビューと最終決定**

1. **レビュー:** プロジェクトオーナーの役割は、戦略・分析AIが作成した『仕様書ドラフト』をレビューし、プロジェクトの目的に合致しているかを\*\*「判断」\*\*することに特化する。  
2. **承認:** 内容が妥当であれば、オーナーはこれを承認し、\*\*『承認済み仕様書』\*\*とする。

### **第5層：実装計画プロセス**

1. **実装計画の依頼:** オーナーは、コーディングAIに『承認済み仕様書』を提示し、これを実現するための具体的な実装計画の策定を依頼する。  
2. **実装計画書の提出:** コーディングAIは、最新のコードベースを考慮し、影響範囲、具体的な実装手順、確認事項などをまとめた\*\*『実装計画書』\*\*を提出する。  
3. **計画の承認:** オーナーは、『実装計画書』の内容を確認し、技術的なアプローチを承認する。このステップが、実装AIの独走を防ぐ重要なチェックポイントとなる。

### **第6層：実装と報告プロセス**

1. **実装:** オーナーが『実装計画書』を承認して初めて、コーディングAIはその計画に沿って安全かつ高品質な実装を開始する。  
2. **報告:** 実装完了後、コーディングAIは成果物（ソースコード）と、変更内容をまとめた\*\*『実装ドキュメント』\*\*を情報ハブに保存・報告する。  
3. **サイクルの開始:** 新たな実装が完了したら、必要に応じて第1層に戻り、次の開発サイクルを開始する。

## **7\. AIへの情報伝達プロトコル**

* **役割の徹底:** 戦略・分析AIには実装方法を求めず、コーディングAIには必ず『承認済み仕様書』をインプットとして与え、『実装計画書』の策定から作業を開始させる。  
* **現状共有の徹底:** 戦略・分析AIとの対話は、(任意で)『AIブリーフィング』による現状共有から始める。  
* **本書による役割認識:** すべてのAIとの対話の冒頭で、本原則に定められた自らの役割を再認識させる。
