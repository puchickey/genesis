# Procedure 01: Checklist Article

## 目的
読者に「自分のことだ」と気づかせ、自然に納得させるチェックリスト型の記事を生成する。

---

## ⚠️ STEP 0: 素材確認（最重要ルール — 記事ごとに必ず実行）

> **このステップを完了するまで、記事を書き始めてはいけない。**
> **複数記事を一括生成する場合も、1記事ずつこのステップを実行する。スキップ厳禁。**

### 0-1. Genesis OS 内の素材調査
まず以下のファイルから、テーマに関連する実体験・事実を検索する:
*   `user_profile.md`（事実・制約・行動履歴）
*   `DailyLog/`（日記・ログ）
*   `note/Note_Project_Setup.md`（戦略・コンセプト）
*   `memories/`（過去の記録）

### 0-2. 素材ヒアリングシートの作成・提示
調査結果を元に、以下のシートを**記事ごとに作成してユーザーに提示する**:

```markdown
## 素材確認シート: [記事タイトル]

### テーマ
（記事の主題を一文で）

### Genesis OS から見つかった素材
- （OS内で確認できた事実・体験を箇条書き）

### 確認が必要な項目（⬜ = 未確認）
1. ⬜ **きっかけ（CAR-C）:** 何がきっかけで始めた/気づいたか？
2. ⬜ **過程・葛藤（CAR-A）:** やってみてどう感じたか？途中の迷いや失敗は？
3. ⬜ **具体的な変化（CAR-R）:** 結果として何がどう変わったか？（数字・時間・感覚）
4. ⬜ **補足:** 他に読者に伝えたいエピソードはあるか？

### 公開価値の判定（ユーザーと合意の上で決定）
- 🟢 GO — 素材が十分にキャッチーで、記事として公開に値する
- 🟡 PENDING — 素材はあるが切り口・深さが弱い。テーマや構成の練り直しが必要
- 🔴 NOGO — この素材では読者に刺さる記事にならない。テーマ変更を推奨
```

### 0-3. 質問回数の制限なし
*   記事のクオリティが上がるのであれば、**質問はいくつ行っても、何回行ってもよい**
*   1回にまとめる必要はない。追加で気になったことがあれば都度聞く
*   ただし質問は**具体的で回答しやすいもの**にする

### 0-4. ユーザー確認と公開判定のルール
*   シートの「確認が必要な項目」について、ユーザーの回答がないまま書き始めてはいけない
*   質問は**最大5問**にまとめて一度に聞く
*   確認後、以下の観点でユーザーと**公開価値を合議**する:
    *   **キャッチーさ:** このテーマ・切り口はターゲット読者の目を引くか？
    *   **深さ:** 表面的な話で終わらず、読者に「なるほど」と思わせる具体性があるか？
    *   **差別化:** 既に世の中にある類似記事と比べて、著者独自の視点があるか？
    *   **再現性:** 読者が読んだ後に「自分もやってみよう」と思える具体策があるか？
*   判定結果:
    *   **🟢 GO:** 執筆に進む
    *   **🟡 PENDING:** 切り口の変更、構成の練り直し、素材の追加ヒアリングを行う
    *   **🔴 NOGO:** この記事はスキップし、別テーマを提案する

### 0-4. 推測禁止の具体ルール
*   **絶対に推測してはいけないもの:**
    *   著者が実際にその行動をしたかどうか
    *   行動中に感じた葛藤や感情の内容
    *   具体的な数値の変化（時間、金額、回数、期間）
    *   ビフォーアフターの差分
*   **OSの記録から引用して良いもの:**
    *   `user_profile.md` に明記されている事実（年齢、職業、資産額、診断名等）
    *   `DailyLog` に記録されている行動履歴
    *   `note/Note_Project_Setup.md` に合意済みの戦略・コンセプト

### 0-5. ヒアリング回答のGenesis OS記録（必須）
*   素材確認や追加質問でユーザーから得た回答は、**記事だけでなくGenesis OS内にも記録する**
*   記録先の優先順位:
    1. `user_profile.md` — 事実・行動パターン・価値観・具体的な運用方法
    2. `note/Note_Project_Setup.md` — 記事戦略・コンセプトに関する決定事項
    3. `DailyLog/` — 時系列で意味のある記録
*   **目的:** 同じ質問を二度しない。一度得た情報はOSの記録として蓄積し、次回以降の記事作成に活用する

### 0-6. 記事コメント・フィードバックのGenesis OS記録（必須）
*   ユーザーが記事ドラフトに対してコメント・修正指示を出した場合、**記事の修正だけでなく、コメントから読み取れる事実・価値観・好みもGenesis OSに記録する**
*   例:
    *   「ここは事実と違う」→ 正しい事実を `user_profile.md` に記録してから記事を修正
    *   「この表現は自分っぽくない」→ 著者のトーン・好みとして記録
    *   「この機能はまだ使ってない」→ 未実施の事実として記録
*   記録先は 0-5 と同じ優先順位（`user_profile.md` → `note/Note_Project_Setup.md` → `DailyLog/`）
*   **手順:** 記録 → 修正の順。先にOSに記録してから記事を直す

---

## 記事の制約
*   **文字数:** 2500〜3000字
*   **トーン:** 共感ベース。押しつけない。「僕の場合は」で語る
*   **核メッセージ:** 全記事共通 → **「不必要なことを見つけて、やめる」**
*   **読者メリット:** 読み終えた後「自分にもやらなくていいことがあるかも」と思えること
*   **著者のスタンス:** HSP気質・元うつ病の自分が、刺激と面倒を減らして生きている事例を提供する。読者ごとに「不必要なもの」は違うことを前提に、押しつけない

## 📌 タイトル設計（最低Aランク以上を必須とする）

### Tier判定基準（らっこ氏 noteタイトルTier表 準拠）

| Tier | パターン | 例 |
|---|---|---|
| **S+** | 数字＋変化型、損失回避型、実証型 | 「30日間で-5kg痩せた全食事メニュー公開」 |
| **S** | ビフォーアフター型、暴露型、問い×答え型 | 「コミュ障だった僕が営業トップになるまでにやったこと」 |
| **A** | 網羅型、具体策＋効果型 | 「初心者が最初の1ヶ月でやるべき筋トレ5ステップ完全ガイド」 |
| B以下 | まとめ型、感想文型、日記型 | 使用禁止 |

### タイトル作成ルール
1. **「結果」を必ず入れる** — 「〇〇した → どうなった」の構造
2. **数字を入れる** — 「5分」「3パターン」「半分」「ゼロ」等
3. **スマホで切れない長さ** — 目安40字以内
4. **読者の「自分ごと感」** — 読者が「自分にも関係ある」と思える入口
5. **B以下のパターン（まとめ、感想、日記、抽象型）は使わない**
6. **損失回避型は控えめに** — 煽りすぎるとトーンが合わない

### セルフチェック
タイトル作成後、以下を確認する：
- [ ] 読者が「結果」を知りたくてクリックする構造になっているか
- [ ] 数字が含まれているか
- [ ] 「〜について」「〜を考えてみた」「〜した話」で終わっていないか

## 記事構成（6部構成・この順序で出力する）

### ① 導入 → 読者の痛み（CAR-C）
*   読者が感じていそうな具体的な場面描写から入る
*   「まだ問題だと気づいていない層」にも届く書き方にする
*   読者が「自分のことだ」と感じる入口
*   結論や解決策は出さない

### ② 気づき → 著者のWHY
*   著者がこの問題に敏感な理由を1〜2文で添える
*   HSP気質・感覚過敏・元うつ病などの文脈を自然に入れる
*   毎回長く書く必要はない。「僕はHSP気質で刺激に弱いから、これが特にきつかった」程度
*   **目的:** 「なぜこの人がここまでやるのか」のWHYを読者に見せる

### ③ チェックリスト（3項目）
*   各項目はYES/NOで直感的に判断できる文にする
*   **「あなたにも、この"不必要"が潜んでいませんか？」** というスタンス
*   断定ではなく「〜しがち」「〜になっていないか」の表現を使う
*   番号付き（①②③）で出力

### ④ チェックリスト項目の展開 → CAR-A（著者の事例）
各項目につき**1〜2パターン**で展開する。以下から選択：

| パターン | 説明 |
|---|---|
| 原因分解型 | なぜ起きるのか |
| 放置リスク提示型 | このままだとどうなりやすいか |
| 誤解解除型 | 多くの人が勘違いしている点 |
| 比較対照型 | 当てはまる人／当てはまらない人 |
| 初期サイン提示型 | まだ軽い段階の兆候 |
| 読者思考の言語化型 | 頭の中の声の代弁 |
| 行動ズレ指摘型 | 努力しているのにズレている点 |
| 構造俯瞰型 | 個人の問題ではなく仕組みの話 |
| 具体例型 | 失敗例、ビフォーアフター、比較例 |

**CAR-Aルール:**
*   展開内に**著者が「不必要だ」と気づいた過程（試行錯誤・葛藤・感情変化）**を含めること
*   「僕の場合は、これをやめた」として添える（読者に押し付けない）
*   **「でもこれは僕にとって不必要だっただけで、あなたには必要かもしれない」** という視点を忘れない
*   **見出しに「（○○型）」と表記しない。** パターン表は内部参照用であり、記事本文には型名を出力しない

### ⑤ AI活用セクション（必須）
*   **「僕はこれをAIにこう聞いて処理している」** を具体的に1〜2例示す
*   著者が実際に試したAIの使い方のみ記載する（推測で書かない）
*   プロンプトや質問文の例を短く添えるとリアリティが出る
*   長くなりすぎない（2〜4文程度）
*   このセクションの趣旨: **「不必要を見極め、やめた後の処理をAIに任せている」** という実践例

### ⑥ まとめ → 核メッセージ + CAR-R
*   著者のビフォーアフター（具体的な変化）を1つ入れること
*   核メッセージに接続する: **「不必要なことを1つ見つけるだけで、生活は変わる」**
*   不安を煽りすぎない。「気づけた時点で前進している」スタンス
*   次の記事への導線を入れる

## CAR + 核メッセージ 品質チェック（出力前セルフレビュー）

出力前に、以下を満たしているか確認する：

| 要素 | チェック |
|---|---|
| **C (Context)** | 読者が「自分に関係ある」と感じる場面描写が導入にあるか |
| **WHY** | 著者がこの問題に向き合う理由（HSP/感覚過敏）が②に入っているか |
| **A (Action)** | 著者が「不必要」を見つけてやめた過程（葛藤・試行錯誤）が展開内にあるか |
| **AI** | 著者が実際に試したAI活用が⑤に入っているか |
| **R (Result)** | 行動の結果（具体的な変化）がまとめにあるか |
| **核メッセージ** | 「不必要なことを見つけてやめる」に接続しているか |

すべて満たしていない場合、**情報不足としてユーザーに質問する。**

## 出力ルール

### note.com Markdown記法（必須）
note.comに対応した記法で出力すること。以下を厳守する:

| 要素 | 記法 | note.comでの挙動 |
|---|---|---|
| 大見出し | `## テキスト` | H2表示。**目次に自動表示される** |
| 小見出し | `### テキスト` | H3表示。**目次に自動表示される** |
| 太字 | `**テキスト**` | 太字表示（強調） |
| 引用 | `> テキスト` | 引用ブロック（核メッセージや印象的な一文に使用） |
| 区切り線 | `---` | セクション間の視覚的な区切り |
| 箇条書き | `- テキスト` | リスト表示（5段階まで） |
| 番号リスト | `1. テキスト` | 番号付きリスト |
| 打ち消し線 | `~~テキスト~~` | 打ち消し線 |

**禁止事項:**
*   `#`（H1）は使用不可（タイトルはnoteのタイトル欄で設定する）
*   `####` 以降はすべてH3に変換されるため使用しない
*   見出しなしの太字だけでセクション分けをしない（目次が生成されない）

**目次:**
*   `##` `###` を使えばnoteが自動で目次を生成する（右サイドバー）
*   記事本文内に目次を挿入したい場合は、noteエディタで「+」→「目次」を使用する
*   ドラフト上では `<!-- noteエディタで貼り付け後、ここに「+」→「目次」を挿入する -->` をコメントとして残す

### 記事構成の記法ガイドライン
*   各セクション（①〜⑥）は `##` で大見出しをつける
*   セクション内のサブトピックには `###` を使う
*   セクション間は `---` で区切る
*   核メッセージや印象的な引用は `>` で引用ブロックにする
*   「やめたことリスト」などの列挙は `- ` で箇条書きにする
*   キーフレーズは `**太字**` で強調する

### 文章スタイル
*   結論を急がない
*   説明口調になりすぎない
*   スマホ閲覧前提で適度に改行（1〜3文ごとに空行）
*   1文は短く
*   **共感を軸にしつつ、読者ごとに「不必要なもの」は違うことを尊重する**
